# Contributing

Thank you for your interest in contributing to Programmable Manufacturing Lab.

This repository is an open community space for Physical AI, physics-informed decision layers, and programmable manufacturing. The goal is to build public-facing, generalized, and synthetic artifacts that help researchers, engineers, practitioners, and builders reason about AI systems for manufacturing decisions.

You do not need private production data or proprietary workflows to contribute.

Small contributions are welcome.

---

## Ways to contribute

Useful contributions include:

- improving documentation
- suggesting readiness criteria
- refining process-mapping templates
- proposing toy benchmarks
- adding synthetic examples
- sharing generalized manufacturing AI failure modes
- improving glossary definitions
- suggesting decision-layer concepts
- proposing uncertainty communication examples
- opening issues with questions or use cases
- commenting on discussions

A useful contribution does not need to be a full implementation. A clear question, critique, example, or improvement can help the project.

---

## Good first contribution types

Good starting points include:

- comment on an existing issue
- suggest one missing field in a template
- improve wording in a document
- propose one synthetic benchmark idea
- add one generalized manufacturing example
- suggest one decision-quality metric
- explain one common manufacturing AI failure mode
- improve one glossary definition

## Contribution principles

### 1. Start from the decision, not the model

This project focuses on manufacturing decision-making.

A contribution is especially useful if it helps answer:

```text
What decision would change because this system exists?
```

---

### 2. Keep examples public-facing and generalized

Please keep examples safe to share publicly.

Good examples are:

- generalized
- educational
- synthetic when possible
- free of private production details
- understandable to researchers, engineers, and builders

---

### 3. Do not share confidential information

Please do not share:

- private production data
- customer or partner information
- sensitive process parameters
- proprietary workflows
- export-controlled or restricted technical details
- information you do not have permission to share

When in doubt, generalize the example or use synthetic data.

---

### 4. Prefer clarity over complexity

This repository is intended to help build shared language.

Simple, clear contributions are valuable.

Examples:

- a better definition
- a clearer table
- a simpler template section
- a practical readiness question
- a synthetic example
- a short discussion prompt

---

### 5. Make contributions useful across domains

Where possible, contributions should be useful beyond one specific process.

For example, instead of writing only about one machine or material, try to describe the broader pattern:

```text
A process may not be AI-ready if the output cannot be linked back to the decision that produced it.
```

---

## Suggested workflow

### For small changes

1. Open an issue or comment on an existing issue.
2. Suggest the change.
3. Submit a pull request if appropriate.

### For documentation changes

1. Find the relevant file.
2. Edit the wording, structure, or examples.
3. Keep the tone practical and public-facing.
4. Submit a pull request.

### For new templates or benchmarks

Before adding a large new artifact, please open an issue describing:

- the purpose
- the intended audience
- the decision problem it supports
- whether it uses synthetic or generalized examples
- how it relates to existing repository artifacts

---

## Pull request checklist

Before submitting a pull request, please check:

- [ ] The contribution is public-facing and safe to share.
- [ ] It does not include private production data.
- [ ] It does not include confidential customer, partner, or workflow details.
- [ ] It is related to Physical AI, manufacturing AI readiness, decision layers, or programmable manufacturing.
- [ ] It is clear enough for people outside your immediate domain to understand.
- [ ] It helps improve shared language, templates, benchmarks, or decision-focused reasoning.

## For manufacturing practitioners

You can contribute without writing code.

Useful practitioner contributions include:

- describing common bottlenecks in general terms
- explaining what makes a process hard to measure
- identifying where AI pilots usually break down
- suggesting what “pilot ready” should mean in practice
- sharing how engineers or operators make decisions today

---

## For AI researchers

Useful AI research contributions include:

- synthetic benchmark ideas
- uncertainty-aware modeling examples
- physics-prior examples
- evaluation metrics
- decision-focused model outputs
- examples where prediction accuracy alone is not enough
